# M7.5 阶段测试列表（M8 E2E 测试基础设施：后端 Seed Hunter 全覆盖，TDD）

> 依据文档：
> - `memory-bank/design/backend_design.md` 第 4 章（4.1~4.9）
> - `memory-bank/implementation-plan.md`（M8）
> 目标：为 M8 阶段 E2E 测试提供可复现基础设施，冻结后端 Seed Hunter 与 REST 种子注入能力测试清单，供后续按测试 ID 执行 Red -> Green。

## 0) 测试环境与执行策略

- 测试分层：
  - Unit：台账契约、匹配判定、hunting 流程、写回语义。
  - API：`POST /api/games/seed-injection` 契约与错误口径。
  - Real-service：启动模式切换与“注入后下一局一次性消费”端到端行为。
- 推荐命令（以实际新增测试文件为准）：
  - Unit：`pytest backend/tests/unit/test_m8_seed_hunt_01_18.py -q`
  - API：`pytest backend/tests/api/games/test_m8_api_seed_injection_19_23.py -q`
  - Real-service：`pytest backend/tests/integration/real_service/test_m8_rs_seed_mode_01_04.py -q`
- 数据隔离约定：
  - 每条用例使用独立临时目录写入 seed catalog JSON。
  - 通过固定 `now_provider` 冻结 `updated_at`，避免时间不稳定。
  - 通过 stub `snapshot_provider` 控制命中/未命中路径，避免依赖随机发牌波动。
- TDD 流程约束：
  1. 人类先指定测试 ID。
  2. 先写测试并执行 Red。
  3. 再做最小实现直到 Green。
  4. 每个测试用例通过后必须回填本文档“执行记录”。

### 1.3 命中判定语义（4.4）

| 测试ID | 测试描述 | 通过条件 |
|---|---|---|
| M7.5-MATCH-01 | `first_turn_seat` 必须精确匹配 | seat 不一致即不命中 |
| M7.5-MATCH-02 | `hands_at_least_by_seat` 采用“至少包含” | 实际手牌可有额外牌，要求牌数量不足时不命中 |
| M7.5-MATCH-03 | 判定仅基于开局可观测状态 | 仅使用开局 snapshot 即可判断，不依赖后续动作推进 |

#### M7.5-MATCH-02 mock 数据形式（用于后续 Red 用例落地）

- mock 输入分两层：
  - 台账约束：`seed_requirement.hands_at_least_by_seat`（最小要求）。
  - 快照输入：`snapshot_provider(seed)` 返回 `hands_by_seat`（实际手牌）。
- seat 键约束：两层都使用 `0/1/2`（JSON 可写为字符串 `"0"|"1"|"2"`）。
- 数量约束：牌数量必须为 `int >= 1`，不允许 `0`/负数/浮点/布尔。

台账片段（最小模板）：

```json
{
  "seed_requirement": {
    "first_turn_seat": 0,
    "hands_at_least_by_seat": {
      "0": {"R_SHI": 1, "B_ZU": 1},
      "1": {"B_XIANG": 1},
      "2": {}
    }
  }
}
```

`snapshot_provider(seed)` 返回值（正例模板，允许额外牌）：

```json
{
  "first_turn_seat": 0,
  "hands_by_seat": {
    "0": {"R_SHI": 1, "B_ZU": 2, "R_MA": 1},
    "1": {"B_XIANG": 1, "R_SHI": 1},
    "2": {"R_JU": 1}
  }
}
```

反例模板（数量不足，应不命中）：

```json
{
  "first_turn_seat": 0,
  "hands_by_seat": {
    "0": {"R_SHI": 1, "R_MA": 1},
    "1": {"B_XIANG": 1},
    "2": {}
  }
}
```

执行断言建议：
- 正例：`_matches_requirement(...) is True`（存在额外牌不影响命中）。
- 反例：`_matches_requirement(...) is False`（任一 seat 任一卡牌数量小于要求即不命中）。

### 1.4 Hunting 流程、写回与退出语义（4.5 / 4.6 / 4.8 / 4.9）

| 测试ID | 测试描述 | 通过条件 |
|---|---|---|
| M7.5-HUNT-01 | `seed_current` 快速验证命中时直接采用 | 命中时不进入区间搜索，结果 seed 为原值 |
| M7.5-HUNT-02 | 快速验证失败后按 `[start,end)` 顺序搜索 | 命中 seed 为区间内首个匹配值 |
| M7.5-HUNT-03 | 搜索耗尽未命中计入失败 | `failed_test_ids` 包含该 case，`case_fail` 正确累加 |
| M7.5-HUNT-04 | 写回字段范围受限 | 仅 `seed_current/updated_at` 被更新，其他字段语义不变 |
| M7.5-HUNT-05 | 写回采用临时文件+原子替换 | 产生 `.tmp` 写入并 replace，避免半写入 |
| M7.5-HUNT-06 | 退出码语义固定 | 全命中时 `run_seed_hunting_mode` 返回 0；任一失败返回非0 |

### 1.5 REST 注入接口与一次性消费（4.7 / 4.9）

| 测试ID | 测试描述 | 通过条件 |
|---|---|---|
| M7.5-API-01 | 合法 seed 注入返回成功并写入运行态 | 返回 200，响应包含 `injected_seed` 与 `apply_scope=next_game_once` |
| M7.5-API-02 | 注入请求体非法返回 400 | `seed` 缺失/类型错误/越界触发参数校验失败 |
| M7.5-API-03 | 重复注入后写覆盖前写 | 后一次注入 seed 在下一局生效 |
| M7.5-API-04 | 注入种子仅下一局消费一次 | 第一场新局消费注入值，后续新局不再沿用该值 |
| M7.5-API-05 | 注入不追溯影响已创建/进行中的对局 | 进行中对局 seed 不变，仅未来新局受影响 |
